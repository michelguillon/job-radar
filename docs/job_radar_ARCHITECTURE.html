What this system is

A personal job search intelligence system that collects job descriptions from public ATS APIs, labels them with a validated 15-field extraction schema, scores them against a candidate profile, and tracks application state. It answers one question: which opportunities are worth pursuing?

Job Radar owns discovery, extraction, scoring, prioritisation, and tracking. cv-tailor owns everything that happens after a role is shortlisted.

Design principles

Eight load-bearing decisions. Each one is a concise statement plus the reasoning that would survive a challenge.

1. Validate the schema on real data before building any automation. The extraction schema was validated on 10 real job descriptions before any collector code was written. It changed 10 times. The most significant change — splitting required_skills into four fields (required_technologies, required_competencies, nice_to_have_technologies, nice_to_have_competencies) — was not foreseeable from first principles. It emerged from labelling JDs that don't use these headings explicitly. A schema designed upfront and validated downstream produces a migration problem. A schema discovered through practice and locked before automation is a design constraint that everything else satisfies.
2. Three schema layers; never mixed. JDRecord is what the JD says — objective, Claude-populated, consistent across all labellers. JobPosting is what the product knows — operational state, system-populated. ApplicationRecord is what Michel thinks — subjective, human-only. Conflating extraction with personal judgement contaminates the training corpus. Conflating product state with extraction creates migration debt. Three layers with named owners keep each concern clean. This is not a preference; it is a correctness requirement for a labelling pipeline that feeds fine-tuning.
3. Score structural fit, not keyword overlap. Most job recommendation systems optimise for keyword overlap — count matching skills, rank accordingly. This fails for non-linear career paths. Job Radar evaluates opportunities in three stages: (1) Structural Fit — does the role match career trajectory, seniority, delivery motion, technical depth, and domain? Produces fit_score. (2) Blocking Constraint Assessment — does the role have hard requirements that prevent success regardless of structural fit? Produces requirement_gaps and blocking_constraints. (3) Opportunity Classification — combines stages 1 and 2 into a plain-English fit_label: strong_fit, good_fit, stretch, blocked_fit, interview_practice, income_bridge. The objective is to identify roles that align with the candidate's trajectory, not to maximise keyword matches.
4. Search mode governs filtering, not scoring. Job searching is not a static optimisation problem. In month 1, structural fit is the primary filter — you want career-defining moves. In month 4, low-fit roles become worth pursuing for interview practice. In month 6 under time pressure, structural misalignment matters less than income continuity. The scorer never changes — only the filter threshold does. search_mode in candidate_profile.yaml controls this: selective (low fit filtered), active (low fit labelled and surfaced separately), broad (all roles shown, fit is context not a gate). Low-fit roles are never silently discarded — they are labelled honestly and surfaced at the appropriate threshold.
5. CLI writes; UI reads. All state changes — collection, labelling, scoring, tracking — go through CLI scripts. The UI reads a pre-built index file. This is not a convenience; it is a correctness guarantee. A UI that writes directly to JSONL files creates concurrent write risks, bypasses the validation pipeline, and makes the CLI an optional interface rather than the single source of truth. The index file is regenerated by the CLI after each pipeline run. The UI is stateless.
6. Batch API as default for offline processing. 200 JDs at ~2,000 tokens each costs roughly half as much via the Batch API as synchronous. The pipeline is not latency-sensitive. Cost is tracked per run, logged to corpus/stats.json. This cost baseline is the input to the fine-tuning budget decision in Phase 6. A pipeline that doesn't track its own costs cannot make a principled argument about when to upgrade from rule-based to fine-tuned scoring. Live verified: 5 records labelled at $0.055 with prompt caching active — schema+examples preamble paid once across the batch.
7. Generate the model prompt from the executable schema, not a copy. The labelling prompt's closed-vocabulary section is generated directly from the models.record enums — add a value to the schema and the prompt updates itself. This eliminates the third copy of the schema (after models/record.py and CORPUS_FINDINGS §1.1) that would otherwise silently rot. The model happily emits enum values the code then rejects if the prompt and schema diverge. Deriving the prompt from the code removes the copy entirely.
8. JSONL files as the universal format. The pipeline produces JSONL at every stage because that is what fine-tuning pipelines consume. Adding a database would require an export step to return to JSONL for Phase 6 — a round-trip that adds complexity without adding capability. JSONL is inspectable with standard Unix tools, appendable without migrations, and versionable with schema_version on every record. Append-only with a version field is a lightweight alternative to database migrations for this access pattern.
9. Job Radar and cv-tailor are separate products with a single integration point. Job Radar answers which opportunities are worth pursuing. cv-tailor answers how to pursue them. These are distinct problems with distinct architectures. The integration is a one-way handoff: a shortlisted role from Job Radar becomes the JD input to cv-tailor. cv-tailor does not feed back into Job Radar. Job Radar is not a networking or contact CRM — that scope is explicitly excluded.
