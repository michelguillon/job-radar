## 1. What Job Radar Is

Job Radar is a personal job search intelligence system. It identifies,
assesses, prioritises, and tracks job opportunities. It answers one
question: **which opportunities are worth pursuing?**

It does not generate application materials. That is cv-tailor's job.

```
Job Radar owns:        cv-tailor owns:
─────────────────      ─────────────────
Discovery              CV tailoring
Extraction             Cover letters
Scoring                Application responses
Prioritisation         Application content
Tracking
```

The integration point is a one-way handoff:

```
Job Radar shortlist
↓
cv-tailor workflow
```

## 3. Architectural Boundaries

### 3.1 — What Job Radar is not

- Not a CV writer (cv-tailor)
- Not a fine-tuning project (Project 5)
- Not an application submission tool
- Not a job board aggregator with user accounts

### 3.2 — Data layer: files all the way down

JSONL files, not a database. No ORM. No migrations. The pipeline
produces JSONL at every stage because that is the format downstream
consumers (fine-tuning, scoring) expect.

The web UI (Phase 5) reads from a pre-built index file generated by
the CLI. It never writes to JSONL directly.

### 3.3 — Extraction vs annotation vs product layer

Three clearly separated schema layers. Each serves a different purpose
and is populated by a different actor.

```
JDRecord          — what the JD says (Claude extracts, objective)
JobPosting        — product metadata (system populates, operational)
ApplicationRecord — personal assessment (Michel populates, subjective)
```

These are never mixed in the same object. `JDRecord` is a pure
extraction artefact. `JobPosting` wraps it with product state.
`ApplicationRecord` tracks personal judgement independently.

### 3.4 — CLI writes, UI reads

All writes go through CLI scripts. The UI is read-only. It never
modifies JSONL directly. The index file (`corpus/index.json`) is
regenerated by `stats.py --export-index` after each pipeline run.

## 4. Schema

### 4.1 — Phase 1 schema (current — locked at v1.2)

`JDRecord` is the extraction schema. It is defined in `models/record.py`
and is the executable source of truth. `CORPUS_FINDINGS.md §1.1` must
stay in sync with it. Any schema change requires both to be updated
and `SCHEMA_VERSION` bumped.

```python
SCHEMA_VERSION = "1.2"

@dataclass
class JDRecord:
    # --- Identity ---
    id: str                      # SHA-256 of normalised text
    source_url: str
    source_ats: str              # "greenhouse"|"lever"|"ashby"|"vc_board"|"manual"
    company: str
    collected_at: str            # ISO datetime
    tier: int                    # 1 / 2 / 3 / 4

    # --- Raw content ---
    raw_html: str | None
    raw_text: str

    # --- Extraction schema (Claude-populated Tier 3+, human-structured Tier 1-2) ---
    role_type: list[str]         # max 3 values
    seniority: str               # "ic"|"senior_ic"|"lead"|"manager"|"director"|"vp"|"exec"
    technical_depth: str         # "hands_on"|"hybrid"|"leadership"
    years_experience_required: str
    required_technologies: list[str]
    required_competencies: list[str]
    nice_to_have_technologies: list[str]
    nice_to_have_competencies: list[str]
    domain: list[str]
    remote_policy: str           # "remote"|"hybrid"|"onsite"|"not_stated"
    location: str                # verbatim from JD
    delivery_motion: list[str]   # "pre_sales"|"direct_delivery"|"partner_delivery"|
                                 # "partner_enablement"|"customer_success"|
                                 # "professional_services"
    leadership_geography: list[str]
    company_size_signal: str     # "startup"|"scale_up"|"enterprise"|"not_stated"
    company_stage: str           # "seed"|"series_a"|"series_b"|"series_c_plus"|
                                 # "pre_ipo"|"listed"|"not_stated"
    culture_signals: list[str]
    raw_observations: str

    # --- Annotation (human-only, Phase 1 temporary home) ---
    # These fields migrate to ApplicationRecord in Phase 2.
    # They remain here for Phase 1 to preserve the 10 existing records.
    fit_score: int | None
    applied: bool
    application_date: str | None
    application_decision: str    # "applied"|"want_to_apply"|"pending"|
                                 # "not_applied_fit"|"not_applied_structural"|
                                 # "not_applied_timing"
    application_decision_notes: str
    location_workable: str       # "yes"|"no"|"conditional"|"unknown"
    location_notes: str
    domain_distance: str         # "low"|"medium"|"high"|"not_assessed"
    blocking_constraints: list[str]
    notes: str
```

### 4.2 — Phase 2 schema additions

These dataclasses are added in Phase 2. They do not affect Phase 1
code. `models/record.py` gains two new classes; `JDRecord` annotation
fields are deprecated (kept for backward compatibility, no longer
populated for new records).

```python
@dataclass
class JobPosting:
    """Product-layer wrapper. Operational metadata about a job opportunity."""
    id: str                      # matches JDRecord.id
    company: str
    role_title: str              # from JDRecord.role_type[0]
    source_url: str
    source_ats: str
    location: str
    remote_policy: str
    date_posted: str | None      # from JD if stated, else None
    first_seen: str              # ISO datetime — when first collected
    last_seen: str               # ISO datetime — when last seen in source
    status: str                  # "active"|"closed"|"unknown"
    extraction: JDRecord         # full extraction artefact

@dataclass
class ApplicationRecord:
    """Personal assessment and workflow state. Human-only. Never Claude."""
    job_id: str                  # matches JobPosting.id
    fit_score: int | None        # 1–10
    priority_score: int | None   # 1–10, separate from fit (see §5.2)
    fit_reasons: str             # free text, written for high-priority roles
    domain_distance: str         # "low"|"medium"|"high"|"not_assessed"
    location_workable: str       # "yes"|"no"|"conditional"|"unknown"
    location_notes: str
    blocking_constraints: list[str]
    application_status: str      # "new"|"review"|"shortlisted"|"applied"|
                                 # "interviewing"|"rejected"|"offer"|"archived"
    application_date: str | None
    notes: str
```

**Why `priority_score` is separate from `fit_score`:**
Fit measures match quality. Priority measures urgency — deadline,
competition, strategic importance, timing. A role can be high fit
but low priority (no rush, always hiring) or lower fit but high
priority (closing soon, dream company). Separating them enables
better sorting in Phase 5.

### 5.1 — What Phase 1 builds

A CLI pipeline: collect → clean → dedupe → label → validate → export.

Each phase produces a durable JSONL checkpoint. The pipeline is
resumable at any point.

```

              → corpus/labelled/failures_{timestamp}.jsonl
export.py     → corpus/finetune_export/export_{timestamp}.jsonl
```

### 5.6 — Tier model

**Tier 1 — Schema-forming (6 JDs — COMPLETE ✅)**
Human labels every field. Claude structures output. Used to discover
and iterate the schema. Contamination rule: Claude extraction runs
only after human labels are saved.

**Tier 2 — Deep review (15 JDs — 10 complete, 5 remaining)**
Claude extracts first. Human does field-by-field review. Schema
stress-tested on harder cases.

**Tier 3 — Light review (50 JDs)**
Claude extracts. Human spends 2–3 minutes checking for systematic
failures only.

**Tier 4 — Automated (130+ JDs, ~200 total)**
Claude extracts. Schema validation in code. No per-JD human review.

**Phase gates:**
- Tier 2 → Tier 3: Claude agreement ≥80% on Tier 2 (excluding
  required/nice-to-have skill fields)
- Tier 3 → Tier 4: no systematic failure mode in Tier 3

### 5.7 — Labelling rules

| Rule | Definition |
|---|---|
| Remote policy default | "based in [city]" = "hybrid". Onsite only if explicit. |
| Location extraction | Verbatim. Never normalise. "not_stated" if absent. |
| Years not stated | Never infer. "not_stated" if not explicit. |
| No external knowledge | Extract only what the JD states. |
| Technology vs competency | Tool/system = technology. Behaviour = competency. |
| Proprietary platforms | Extract as stated, note in raw_observations. |
| company_stage | Explicit or unambiguously inferable only. |
| Sparse culture signals | Do not pad. Sparse list is valid signal. |
| Nice-to-have | Only if JD explicitly signals optionality. Never infer. |
| delivery_motion | How value is delivered, not what function. Multi-value ok. |
| leadership_geography | Empty [] for IC roles or scope not stated. |
| Contamination rule | Claude extraction on Tier 1 JDs runs after human labels only. |

## 6. Phase 2 — Scoring Engine

### 6.1 — Purpose

Consumes `JDRecord` extraction fields and a candidate profile to
produce fit and priority scores per role. Initial implementation is
rule-based — no fine-tuning required.

### 6.2 — Candidate profile

A structured YAML file (`candidate_profile.yaml`) defining Michel's
background against which every JD is scored.

```yaml
candidate:
  name: Michel Guillon
  target_roles:
    - Solutions Engineering
    - Solutions Architecture
    - Solutions Consulting
    - AI Delivery
    - Partner SA
  target_seniority:
    - director
    - manager
    - lead
    - senior_ic
  target_domains:
    - AI Platform
    - AI/ML
    - Enterprise Software
    - FinTech
    - AdTech
  technical_depth: hybrid
  location: London
  acceptable_remote_policy:
    - hybrid
    - remote
  domain_strengths:
    - AdTech
    - AI/ML
    - Enterprise Software
  domain_gaps:
    - Customer Experience
    - Revenue Technology
  skills:
    technologies:
      - Python
      - LLM APIs
      - AI/ML frameworks
      - RAG pipelines
      - API platforms
    competencies:
      - pre-sales team leadership
      - enterprise sales cycle management
      - executive communication
      - solution design and architecture
      - GTM strategy ownership
      - EMEA multi-regional management
```

### 6.3 — Scoring dimensions

Rule-based scoring across 5 dimensions. Each dimension scores 0–2.
Total raw score: 0–10. Normalised to fit_score: 1–10.

| Dimension | Weight | Logic |
|---|---|---|
| role_match | 2 | JD role_type intersects target_roles |
| seniority_match | 2 | JD seniority in target_seniority |
| technical_depth_match | 2 | JD technical_depth matches candidate profile |
| domain_match | 2 | JD domain intersects target_domains; domain_strengths bonus |
| location_match | 2 | JD location + remote_policy vs candidate location rules |

`priority_score` adds urgency signals on top of fit:
- `company_stage` (startup/scale-up = higher urgency)
- `application_decision` history (similar roles applied = calibration)
- `blocking_constraints` (any = reduce priority)

### 6.4 — Output

`ApplicationRecord` is created for every scored role:

```
corpus/scored/scored_{timestamp}.jsonl
```

One `ApplicationRecord` per `JobPosting`. Initial
`application_status: "new"` for all. Human reviews and updates via
CLI or UI filter.

### 6.5 — Implementation

New files:
- `candidate_profile.yaml` — candidate definition

- `score.py` — CLI entry point


## 8. Phase 4 — Discovery Layer

### 8.1 — Purpose

Adds continuous, automated discovery. Surfaces new relevant roles
without requiring manual source management.

### 8.2 — Components

**Scheduled collection:** `collect.py --source all` runs weekly via
cron. New records appended, not overwritten. Dedup prevents
re-processing.

**Incremental scoring:** New records scored automatically after
collection. `ApplicationRecord` created with `application_status:
"new"` and `application_decision: "pending"`.

**Daily digest:** `digest.py` generates a summary of new roles
above a fit threshold.

```bash
python digest.py --min-fit 6 --since yesterday
```

Output: terminal summary + optional export to `corpus/digest_{date}.md`

**New source candidates** (evaluate before adding):
- LinkedIn job alerts via email → parse digest emails
- RSS feeds from company career pages
- Google Alerts for target companies + "hiring"

## 9. Phase 5 — UI

### 9.1 — Purpose

Read-only interface for browsing, filtering, and inspecting the corpus.
Serves Michel the job seeker, not Michel the data engineer.

### 9.2 — What the UI shows

- **Browse view** — filterable table: company, role title, fit score,
  priority score, location, application status, date first seen
- **Detail view** — full record: all extraction fields, fit reasons,
  blocking constraints, culture signals, full JD text, source URL
- **Pipeline view** — grouped by application_status, sorted by
  priority_score
- **Corpus stats** — record count, cost to date, coverage, dedup rate

### 9.3 — What the UI never does

- Write to any JSONL file
- Trigger collection or labelling
- Edit records
- Submit applications

### 9.4 — Architecture

**Static index approach:**

`python stats.py --export-index` generates `corpus/index.json` — a
flat denormalised array of all validated records. The UI reads this
one file. No API, no database, no additional backend service.

The UI is a single HTML file with vanilla JavaScript. Deployable
anywhere. No build step, no framework.

**Filters available:**
- fit_score (min/max)
- priority_score (min/max)
- application_status (multi-select)
- location_workable (yes/no/conditional/unknown)
- domain (multi-select)
- role_type (multi-select)
- seniority (multi-select)
- company_stage (multi-select)
- date_first_seen (range)

**Docker service:**
```yaml
ui:
  image: nginx:alpine
  volumes:
    - ./ui:/usr/share/nginx/html
    - ./corpus/index.json:/usr/share/nginx/html/data/index.json
  ports:
    - "8080:80"
```

New files:

## 10. Phase 6 — Fine-Tuned Analyser (Future Enhancement)

This phase is **explicitly deferred**. It becomes Project 5 when the
corpus is large enough to justify fine-tuning (target: 500+ labelled
records with human-validated eval set).

**What it does:** Replaces the rule-based scoring engine (Phase 2)
with a fine-tuned model that predicts fit_score and blocking_constraints
from raw JD text. The corpus built in Phase 1 is the training data.
The eval set from Phase 1 is the evaluation set.

**Why deferred:** Fine-tuning a domain-specific model on <200 records
produces marginal gains over a well-designed rule-based scorer. The
rule-based scorer in Phase 2 is good enough to be useful immediately.
Fine-tuning becomes valuable when the corpus is large and the
rule-based scorer's failure modes are well-understood.

**Handoff to Project 5:**
- `corpus/finetune_export/export_train_*.jsonl`
- `corpus/finetune_export/export_eval_*.jsonl`
- `CORPUS_FINDINGS.md` — schema, labelling rules, failure modes
- `corpus/stats.json` — cost baseline

## 11. Relationship to cv-tailor

cv-tailor is live in production. It is not modified as part of this
project.

**Current state:** Both systems perform JD analysis independently.
cv-tailor's `JDAnalysis` is a flat extraction. Job Radar's `JDRecord`
is richer and more structured. They produce overlapping but
non-contradictory outputs for the same JD — Michel is the
consistency layer until a future integration is designed.

**Integration point (future, not in scope):**

```
Job Radar shortlist (fit_score ≥ 7, application_status: "shortlisted")
↓
cv-tailor workflow
↓
Tailored CV + cover letter
```

The integration is one-way: Job Radar tells cv-tailor which role to
tailor for. cv-tailor does not feed back into Job Radar.

A deeper integration (Job Radar providing pre-computed JD analysis to
cv-tailor, skipping cv-tailor's Phase 0 extraction) is possible but
deferred until both systems are stable and usage patterns are
understood.

## 12. Architecture Decisions

### 12.1 — Job Radar as one product, not separate projects

**Decision:** Phases 1–5 are one product. Phase 6 is Project 5.

**Rationale:** The corpus builder alone is not useful to Michel the
job seeker. The job search tool needs the corpus. Building them
separately creates a handoff problem and two incomplete products.
One product with phases produces value incrementally.

### 12.2 — Rule-based scoring before fine-tuning

**Decision:** Phase 2 is rule-based. Fine-tuning is Phase 6 / Project 5.

**Rationale:** 200 labelled records is insufficient for meaningful
fine-tuning of a domain-specific scorer. A well-designed rule-based
scorer on a structured schema produces comparable results immediately,
without GPU, without training cost, and without the data requirements
of fine-tuning. Fine-tuning adds value after the corpus exceeds 500
records and the rule-based scorer's failure modes are documented.

### 12.3 — Three-layer schema separation

**Decision:** JDRecord (extraction) / JobPosting (product) /
ApplicationRecord (annotation) are separate dataclasses.

**Rationale:** Mixing extraction labels with personal judgement
contaminates the training corpus. Mixing product state with extraction
creates migration problems when the product evolves. Three layers
with clear ownership (Claude / system / Michel) keep each concern
clean.

### 12.4 — Static index for UI

**Decision:** UI reads `corpus/index.json`, not a live API.

**Rationale:** The UI is read-only. A live API adds a backend service,
a port, auth, and operational complexity for no functional benefit.
A pre-built index regenerated by CLI after each pipeline run is
simpler, faster to render, and consistent with "files all the way
down."

### 12.5 — CLI writes, UI reads

**Decision:** All writes go through CLI. UI never modifies state.

**Rationale:** Preserves the CLI as the single source of truth.
Prevents accidental writes from the UI corrupting JSONL files.
Keeps the UI stateless and deployable anywhere.

## 14. Interview Narrative

**One product, multiple learning layers.** Job Radar is not a demo
project. It is a system Michel uses daily for his job search. This
means it has real requirements, real failure modes, and real
constraints — not invented ones. That authenticity shows in the
architecture decisions.

**Schema as the hardest problem.** The extraction schema was validated
on 10 real JDs before any code was written. It changed 10 times.
The most significant changes — splitting required_skills into four
fields, separating extraction from annotation, adding delivery_motion
as a first-class dimension — were not foreseeable from first principles.
They emerged from practice. That is how real data engineering works.

**Three-layer schema separation.** The JDRecord / JobPosting /
ApplicationRecord split demonstrates understanding of the difference
between what data says, what a product does with it, and what a user
thinks about it. Most candidates conflate these. The ability to name
the boundary and enforce it structurally is a differentiator.

**Incremental value delivery.** Phase 1 produces a working corpus.
Phase 2 produces actionable fit scores. Phase 3 replaces a
spreadsheet. Each phase is independently valuable. This is how
production systems get built — not in one big release.

**Cost awareness from the start.** Batch API chosen in Phase 1 before
a single labelling run. Cost tracked per run. Fine-tuning deferred
until the corpus justifies it. These are the decisions of someone
who has thought about production constraints, not someone building
a demo.
