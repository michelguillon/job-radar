## 1. What Job Radar Is

Job Radar is a personal job search intelligence system. It identifies,
assesses, prioritises, and tracks job opportunities. It answers one
question: **which opportunities are worth pursuing?**

It does not generate application materials. That is cv-tailor's job.
It does not manage networking contacts or relationships. That is a CRM's job.

```
Job Radar owns:        cv-tailor owns:
‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ      ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ

```

The integration point is a one-way handoff:


## 3. Architectural Boundaries

### 3.1 ‚Äî What Job Radar is not

- Not a CV writer (cv-tailor)
- Not a fine-tuning project (Project 5)
- Not an application submission tool
- Not a job board aggregator with user accounts

### 3.2 ‚Äî Data layer: files all the way down

JSONL files, not a database. No ORM. No migrations. The pipeline
produces JSONL at every stage because that is the format downstream
consumers (fine-tuning, scoring) expect.

The web UI (Phase 5) reads from a pre-built index file generated by
the CLI. It never writes to JSONL directly.

### 3.3 ‚Äî Extraction vs annotation vs product layer

Three clearly separated schema layers. Each serves a different purpose
and is populated by a different actor.

```
JDRecord          ‚Äî what the JD says (Claude extracts, objective)
JobPosting        ‚Äî product metadata (system populates, operational)
ApplicationRecord ‚Äî personal assessment (Michel populates, subjective)
```

These are never mixed in the same object. `JDRecord` is a pure
extraction artefact. `JobPosting` wraps it with product state.
`ApplicationRecord` tracks personal judgement independently.

### 3.4 ‚Äî CLI writes, UI reads

All writes go through CLI scripts. The UI is read-only. It never
modifies JSONL directly. The index file (`corpus/index.json`) is
regenerated by `stats.py --export-index` after each pipeline run.

## 4. Schema

### 4.1 ‚Äî Phase 1 schema (current ‚Äî locked at v1.2)

`JDRecord` is the extraction schema. It is defined in `models/record.py`
and is the executable source of truth. `CORPUS_FINDINGS.md ¬ß1.1` must
stay in sync with it. Any schema change requires both to be updated
and `SCHEMA_VERSION` bumped.

```python
SCHEMA_VERSION = "1.2"

@dataclass
class JDRecord:
    # --- Identity ---
    id: str                      # SHA-256 of normalised text
    source_url: str
    source_ats: str              # "greenhouse"|"lever"|"ashby"|"vc_board"|"manual"
    company: str
    collected_at: str            # ISO datetime
    tier: int                    # 1 / 2 / 3 / 4

    # --- Raw content ---
    raw_html: str | None
    raw_text: str

    # --- Extraction schema (Claude-populated Tier 3+, human-structured Tier 1-2) ---
    role_type: list[str]         # max 3 values
    seniority: str               # "ic"|"senior_ic"|"lead"|"manager"|"director"|"vp"|"exec"
    technical_depth: str         # "hands_on"|"hybrid"|"leadership"
    years_experience_required: str
    required_technologies: list[str]
    required_competencies: list[str]
    nice_to_have_technologies: list[str]
    nice_to_have_competencies: list[str]
    domain: list[str]
    remote_policy: str           # "remote"|"hybrid"|"onsite"|"not_stated"
    location: str                # verbatim from JD
    delivery_motion: list[str]   # "pre_sales"|"direct_delivery"|"partner_delivery"|
                                 # "partner_enablement"|"customer_success"|
                                 # "professional_services"
    leadership_geography: list[str]
    company_size_signal: str     # "startup"|"scale_up"|"enterprise"|"not_stated"
    company_stage: str           # "seed"|"series_a"|"series_b"|"series_c_plus"|
                                 # "pre_ipo"|"listed"|"not_stated"
    culture_signals: list[str]
    raw_observations: str

    # --- Annotation (human-only, Phase 1 temporary home) ---
    # These fields migrate to ApplicationRecord in Phase 2.
    # They remain here for Phase 1 to preserve the 10 existing records.
    fit_score: int | None
    applied: bool
    application_date: str | None
    application_decision: str    # "applied"|"want_to_apply"|"pending"|
                                 # "not_applied_fit"|"not_applied_structural"|
                                 # "not_applied_timing"
    application_decision_notes: str
    location_workable: str       # "yes"|"no"|"conditional"|"unknown"
    location_notes: str
    domain_distance: str         # "low"|"medium"|"high"|"not_assessed"
    blocking_constraints: list[str]
    notes: str
```

### 4.2 ‚Äî Phase 2 schema additions

These dataclasses are added in Phase 2. They do not affect Phase 1
code. `models/record.py` gains two new classes; `JDRecord` annotation
fields are deprecated (kept for backward compatibility, no longer
populated for new records).

```python
@dataclass
class JobPosting:
    """Product-layer wrapper. Operational metadata about a job opportunity."""
    id: str                      # matches JDRecord.id
    company: str
    role_title: str              # from JDRecord.role_type[0]
    source_url: str
    source_ats: str
    location: str
    remote_policy: str
    date_posted: str | None      # from JD if stated, else None
    first_seen: str              # ISO datetime ‚Äî when first collected
    last_seen: str               # ISO datetime ‚Äî when last seen in source
    status: str                  # "active"|"closed"|"unknown"
    extraction: JDRecord         # full extraction artefact

@dataclass
class ApplicationRecord:
    """Personal assessment and workflow state. Human-only. Never Claude."""
    job_id: str                  # matches JobPosting.id
    fit_score: int | None        # 1‚Äì10
    priority_score: int | None   # 1‚Äì10, separate from fit (see ¬ß6.3)
    fit_label: str               # "strong_fit"|"good_fit"|"stretch"|
                                 # "blocked_fit"|"interview_practice"|
                                 # "income_bridge"
    fit_label_reason: str        # one sentence plain-English interpretation
    fit_reasons: str             # free text, written for high-priority roles
    requirement_gaps: list[str]  # capabilities that may reduce competitiveness
                                 # but do not prevent application
    domain_distance: str         # "low"|"medium"|"high"|"not_assessed"
    location_workable: str       # "yes"|"no"|"conditional"|"unknown"
    location_notes: str
    blocking_constraints: list[str]  # hard stops that likely prevent success
    application_status: str      # "new"|"review"|"shortlisted"|"applied"|
                                 # "interviewing"|"rejected"|"offer"|"archived"
    application_date: str | None
    outcome: str | None          # None until terminal state reached
                                 # "rejected_pre_screen"|"rejected_post_screen"|
                                 # "rejected_interview"|"rejected_final"|
                                 # "offer_declined"|"offer_accepted"|"withdrew"
    outcome_notes: str
    notes: str
```

**Why `priority_score` is separate from `fit_score`:**
Fit measures match quality. Priority measures urgency ‚Äî deadline,
competition, strategic importance, timing. A role can be high fit
but low priority (no rush, always hiring) or lower fit but high
priority (closing soon, dream company). Separating them enables
better sorting in Phase 5.

### 5.6 ‚Äî Tier model

**Tier 1 ‚Äî Schema-forming (6 JDs ‚Äî COMPLETE ‚úÖ)**
Human labels every field. Claude structures output. Used to discover
and iterate the schema. Contamination rule: Claude extraction runs
only after human labels are saved.

**Tier 2 ‚Äî Deep review (15 JDs ‚Äî 10 complete, 5 remaining ‚Äî resume in Phase 2 prep)**
Claude extracts first. Human does field-by-field review. Schema
stress-tested on harder cases.

**Tier 3 ‚Äî Light review (50 JDs)**
Claude extracts. Human spends 2‚Äì3 minutes checking for systematic
failures only.

**Tier 4 ‚Äî Automated (130+ JDs, ~200 total)**
Claude extracts. Schema validation in code. No per-JD human review.

**Phase gates:**
- Tier 2 ‚Üí Tier 3: Claude agreement ‚â•80% on Tier 2 (excluding
  required/nice-to-have skill fields)
- Tier 3 ‚Üí Tier 4: no systematic failure mode in Tier 3

### 5.7 ‚Äî Labelling rules

| Rule | Definition |
|---|---|
| Remote policy default | "based in [city]" = "hybrid". Onsite only if explicit. |
| Location extraction | Verbatim. Never normalise. "not_stated" if absent. |
| Years not stated | Never infer. "not_stated" if not explicit. |
| No external knowledge | Extract only what the JD states. |
| Technology vs competency | Tool/system = technology. Behaviour = competency. |
| Proprietary platforms | Extract as stated, note in raw_observations. |
| company_stage | Explicit or unambiguously inferable only. |
| Sparse culture signals | Do not pad. Sparse list is valid signal. |
| Nice-to-have | Only if JD explicitly signals optionality. Never infer. |
| delivery_motion | How value is delivered, not what function. Multi-value ok. |
| leadership_geography | Empty [] for IC roles or scope not stated. |
| Contamination rule | Claude extraction on Tier 1 JDs runs after human labels only. |

### 5.8 ‚Äî Pre-label filter (added Phase 3)

The first real collection pulled **2,507** global postings; labelling all of them

- **Location screen** ‚Äî keep UK / London / UK-remote / Europe-remote / EMEA-remote /

### 5.10 ‚Äî GTM / partner observation watchlist (no scoring impact)

## 6. Phase 2 ‚Äî Scoring Engine

### 6.1 ‚Äî Purpose

Consumes `JDRecord` extraction fields and a candidate profile to
produce fit scores, requirement gap assessments, fit labels, and
priority scores per role. Initial implementation is rule-based ‚Äî
no fine-tuning required.

### 6.2 ‚Äî Design principle: structural fit before requirement matching

Most job recommendation systems optimise for keyword overlap. They
compare a CV against a JD, count matching skills or titles, and rank
accordingly. This works for linear career paths. It performs poorly
for candidates whose careers span multiple disciplines, where the
strongest opportunities are defined by patterns of experience rather
than shared keywords.

Job Radar is designed around a different assumption:

> A good opportunity is one that fits the candidate's trajectory,
> not simply their vocabulary.

The scorer evaluates opportunities in three stages:

---

**Stage 1 ‚Äî Structural Fit Assessment**

Measures whether the role aligns with the candidate's broader career
pattern. Produces `fit_score: int` (1‚Äì10).

Structural fit considers:
- role alignment (`role_type`)
- seniority alignment (`seniority`)
- delivery motion alignment (`delivery_motion`)
- technical depth alignment (`technical_depth`)
- domain alignment (`domain`)
- leadership pattern alignment

The fit score deliberately ignores individual requirements and focuses
on overall alignment. A role may have excellent structural fit even if
one or more requirements are currently unmet.

---

**Stage 2 ‚Äî Blocking Constraint Assessment**

Identifies specific requirements that may prevent success even when
structural fit is high. Produces `requirement_gaps: list[str]` and
`blocking_constraints: list[str]`.

```
requirement_gap
    A missing capability that may reduce competitiveness.
    Example: "Preferred Salesforce experience"

blocking_constraint
    A requirement that is likely to prevent success regardless
    of structural fit.
    Examples: "Native German speaker required"
              "M&A transaction leadership required"
              "Active security clearance required"
```

---

**Stage 3 ‚Äî Opportunity Classification**

The user-facing output is not a raw score. It is an interpretation.
Combines structural fit and requirement assessment to produce
`fit_label: str` and `fit_label_reason: str`.

```python
fit_label: str
# "strong_fit"         ‚Äî high structural fit, no blocking constraints
# "good_fit"           ‚Äî solid structural fit, minor gaps
# "stretch"            ‚Äî moderate structural fit, meaningful gaps
# "blocked_fit"        ‚Äî high structural fit, blocking constraint present;
#                        structurally excellent but not viable currently
# "interview_practice" ‚Äî low structural fit, no blocking constraints;
#                        worth pursuing for experience, not a career move
# "income_bridge"      ‚Äî low structural fit; apply under time pressure only

fit_label_reason: str  # one sentence shown in UI
# e.g. "Strong role and seniority match, adjacent domain ‚Äî high conviction."
# e.g. "Strong structural fit but lacks required M&A experience."
# e.g. "Moderate structural fit with multiple domain gaps ‚Äî useful practice."
# e.g. "Weak structural fit ‚Äî good interview experience, unlikely career move."
```

---

**Core principle:**

> Identify opportunities that align with the candidate's career
> trajectory, explain any meaningful gaps, and present those
> opportunities in a way that supports good decision making.

Keyword matching is useful evidence, but it is not the objective.
Keyword overlap is a byproduct of structural alignment, not the
signal itself.

---

### 6.3 ‚Äî Search mode: dynamic fit filtering

The scoring system remains constant. What changes over time is how
opportunities are filtered and presented, controlled by `search_mode`
in `candidate_profile.yaml`.

```python
search_mode: str
# "selective" ‚Äî high-conviction opportunities only; low structural fit
#               roles filtered out
# "active"    ‚Äî stretch opportunities visible but clearly labelled and
#               separated from core targets
# "broad"     ‚Äî all opportunities shown; structural fit and requirement
#               gaps are context, not filters; use under time pressure
```

**Why this exists:** Job searching is not a static optimisation
problem. In month 1, structural fit is the primary filter ‚Äî you are
looking for career-defining moves. In month 4, low structural fit
roles become worth pursuing for interview practice. In month 6, with
urgency rising, structural misalignment matters less than income
continuity. The scorer never changes ‚Äî only the filter threshold does.

**Filter behaviour by mode:**

| fit_label | selective | active | broad |
|---|---|---|---|
| strong_fit | ‚úÖ shown | ‚úÖ shown | ‚úÖ shown |
| good_fit | ‚úÖ shown | ‚úÖ shown | ‚úÖ shown |
| stretch | ‚úÖ shown | ‚úÖ shown | ‚úÖ shown |
| blocked_fit | ‚úÖ shown | ‚úÖ shown | ‚úÖ shown |
| interview_practice | ‚ùå filtered | ‚öÝÔ∏è separate section | ‚úÖ shown |
| income_bridge | ‚ùå filtered | ‚ùå filtered | ‚öÝÔ∏è separate section |

Low structural fit roles are never silently discarded ‚Äî they are
labelled honestly and surfaced at the appropriate mode threshold.

### 6.5 ‚Äî Scoring dimensions

Rule-based scoring across 5 structural dimensions. Each scores 0‚Äì2.
Total raw score: 0‚Äì10. Normalised to `fit_score`: 1‚Äì10.

| Dimension | Weight | Logic |
|---|---|---|
| role_match | 2 | JD role_type intersects target_roles |
| seniority_match | 2 | JD seniority in target_seniority |
| technical_depth_match | 2 | JD technical_depth matches candidate profile |
| domain_match | 2 | JD domain intersects target_domains; domain_strengths bonus |
| location_match | 2 | JD location + remote_policy vs candidate rules |

`priority_score` adds urgency signals on top of fit:
- `company_stage` (startup/scale-up = higher urgency)
- `blocking_constraints` (any = reduce priority)
- `search_mode` (broad mode elevates lower-fit roles)

### 6.9 ‚Äî Scoring model refinement: gates vs signal, and calibration

The flat ¬ß6.5 model (5 equal-weight 0‚Äì2 dimensions, summed) was found, on the
first real run, to have almost no resolution: on a curated corpus the seniority
and location dimensions saturated at maximum for every realistic JD, so four of
five dimensions did no discriminating work and nearly everything became
`strong_fit`. Two corrections:

**Gates vs signal (Stage 1).** Dimensions are split by role:
- **Signal** sets the 0‚Äì10 scale ‚Äî `role` (√ó2), `domain` (√ó2), `technical_depth`
  (√ó1). These are what actually differentiate fit.
- **Gates** are table stakes that can only *subtract* ‚Äî `seniority` (binary,
  in-band or a penalty; no partial credit) and `location` (pass / unclear / fail
  penalties). A gate hit contributes nothing; a gate miss pulls the score down.
  This stops table-stakes dimensions from inflating every score.

**Stage 2 can override Stage 1 (capability blocker).** Structural enum match can
be high while a role is infeasible because it mandates hands-on specialist skills
the candidate lacks. When the candidate is not a hands-on specialist, the JD
demands `hands_on` depth, and a cluster (‚â• threshold) of *required* technologies
are unmet by the candidate's proficient skills, the scorer emits a
`blocking_constraint` that demotes the label (e.g. `good_fit` ‚Üí `blocked_fit`).
This is what makes requirement assessment dominate a misleadingly high structural
score, not merely annotate it.

#### Design principle ‚Äî the negative-calibration corpus

A scorer is only as trustworthy as the examples that prove it discriminates. It is
not enough to confirm that good roles score well ‚Äî the corpus must include
**negative examples**: roles that *should* score low, or that are structurally
attractive but genuinely infeasible (the Databricks case ‚Äî strong SA/AI-Platform
enums, but mandatory hands-on Spark/SQL/Databricks/multi-cloud). These negatives
are added to the corpus **specifically to validate discrimination and to set
thresholds from evidence rather than assumption.** A threshold or penalty
magnitude is calibrated against where real positive and negative examples actually
fall, then frozen with a test that pins the calibration anchor (e.g. "Databricks
must be `blocked_fit`").

Most rule-based scoring skips this and tunes only against positives, which is how
"everything is a strong fit" ships. Penalty magnitudes, the unmet-requirements
threshold, and the `fit_label` thresholds remain **provisional** until the
negative-calibration corpus is in place. (Implemented: `scoring/CLAUDE.md`.)


- recommendation quality assessment ‚Äî is structural fit predicting success?
- future fine-tuning dataset ‚Äî real outcome labels on scored JDs

Implementation is lightweight: `outcome` is an **event in the activity log**
(¬ß7.4), *derived* into live state at read time ‚Äî **not** a field on

Full calibration analysis deferred until enough outcomes accumulate (target:
20+ completed applications) ‚Äî build the capture now, not the analysis.

```python
# OUTCOME (models/record.py) ‚Äî value of an `outcome` activity-log event:
# "rejected_pre_screen" | "rejected_post_screen" | "rejected_interview" |
# "rejected_final" | "offer_declined" | "offer_accepted" | "withdrew"
# outcome_notes is carried in the event's `notes`.
```

### 7.4 ‚Äî Implementation (`track.py`, model C)

**The design problem.** `score.py` regenerates every `ApplicationRecord` from
scratch on each run ‚Äî the scorer is pure and always emits
`application_status="new"`, `notes=""`. So workflow state written *into* a scored
record would be wiped by the next collection‚Üílabel‚Üíscore cycle.

**Resolution ‚Äî model C (append-only event log as source of truth)**
(forks resolved in `job_radar_TRACKER_PLAN.md`; superseded the earlier "updates
ApplicationRecord in corpus/scored/" sketch ‚Äî see CLAUDE.md deviation 23):

- `corpus/activity_log.jsonl` is the **single source of truth** for workflow
  state. `track.py` only ever **appends** events; it never edits a scored file,
  never touches the scorer. Re-scoring is therefore always safe.
- A job's **live state** = its latest score (regenerable) **joined** with a
  **projection** folded from the log by `job_id`.
- `outcome` / `application_date` are **derived** at read time (Log-only fork) ‚Äî
  `ApplicationRecord` and `SCHEMA_VERSION` (1.3) are untouched.

**Event format** (`corpus/activity_log.jsonl`, append-only, never edited):
```json
{"v": 1, "ts": "2026-06-10T14:30:00Z", "job_id": "sha256:abc",
 "event": "status", "value": "applied", "notes": "referral"}
```
`event ‚àà {status, outcome, note, title}` (`models.record.ACTIVITY_EVENT`); a
`status` value is an `APPLICATION_STATUS` (minus the implicit `new`), an `outcome`
value is an `OUTCOME`, a `title` value is a non-empty display-title override, a
`note` carries text in `notes` with `value: null`. `validate_activity_event`
guards the closed vocab before any line is written.

**Projection** (fold a job's events in `ts` order): `status` = latest status
event (default `new`); `outcome` = latest outcome event (default `None`);
`application_date` = date of the *earliest* `status=applied` event;
`title_override` = latest title event (default `None`); `notes` = most recent
non-empty note.

**Title resolution** (presentation only, never scored): human override (a `title`
event) ‚Üí sidecar title (by `source_url`) ‚Üí `raw_text` first line ‚Üí `job_id`. The
override exists because the schema-locked JDRecord has no title field and the
sidecar collides on legacy `source_url="unknown"` records. Set with
`track.py --job-id ‚Ä¶ --title "‚Ä¶"`.

**Stable join key:** `job_id` is the JD content hash. If a JD's text changes it
gets a *new* hash ‚Üí a new posting revision ‚Üí workflow does not carry across. This
is accepted behaviour, not a bug.

**CLI:**
```bash
python -m cli.track --job-id sha256:abc --status applied
python -m cli.track --job-id sha256:abc --status interviewing --notes "First round booked"
python -m cli.track --job-id sha256:abc --outcome rejected_post_screen
python -m cli.track --job-id sha256:abc --title "Solutions Engineer"   # display override
python -m cli.track --job-id sha256:abc --notes "recruiter emailed"    # pure note
python -m cli.track list --status shortlisted
python -m cli.track list --min-fit 7 --location-workable yes
```
One CLI call can record several events (e.g. `--status` + `--title`); they share a
`ts` and `--notes` attaches to the first.
- Writing an unknown `job_id` (not in any scored file) errors unless `--force`
  (orphan-event guard).
- Status transitions are **forgiving** ‚Äî an out-of-order move warns but never
  blocks (a job search skips and backtracks stages).
- `list` joins score + JD + sidecar title + workflow and prints a table sorted by
  `priority_score` desc, showing **all** labels. `--location-workable` is a
  **coarse, sidecar-derived** read-only signal (profile London base / remote
  acceptable / no relocation) ‚Äî `ApplicationRecord` has no location field and the
  JDRecord one is a legacy stub, so this is **not** a scoring change.

## 8. Phase 4 ‚Äî Discovery Layer

### 8.1 ‚Äî Purpose

Adds continuous, automated discovery. Surfaces new relevant roles
without requiring manual source management.

### 8.2 ‚Äî Components

**Scheduled collection:** `collect.py --source all` runs weekly via
cron. New records appended, not overwritten. Dedup prevents
re-processing.

**Incremental collection (built ‚Äî `collect.py`):** every run after the first
collects only jobs new/updated since the last successful run, cutting the count
entering the **paid** downstream pipeline (Batch labelling) from O(all live jobs)
to O(new). The public board APIs expose **no server-side date filter**
(Greenhouse's `updated_after` is Harvest-API-only; Lever/Ashby boards take none),
so filtering is **client-side** on each job's own timestamp ‚Äî the real saving is
downstream, not the single bulk GET.

- **Per-source cursor** `corpus/.last_collected_{source}` (gitignored) holds the
  **start** timestamp of the last successful run. Start-not-finish guarantees a
  job updated *during* a run is re-collected next time, never skipped. No cursor ‚Üí
  full collection (first run, or a new source).
- **Capability by source** (see `collectors/CLAUDE.md` for the matrix):
  greenhouse filters on `updated_at` (new + edited); ashby on `publishedAt` (new
  only ‚Äî no `updatedAt` on the public feed; a periodic `--full` reconciles edits);
  **lever has no timestamp at all ‚Üí always full collection** (relies on dedupe).
  `INCREMENTAL_SOURCES` is derived from each collector's `SUPPORTS_INCREMENTAL`.
- **`--full`** ignores cursors and re-fetches everything (after a schema change or
  for debugging), then re-baselines the cursor to now.
- **Advance rules:** a cursor advances only on a full-source run (no `--company`)
  and only for a source that returned ‚â•1 job, so a `--company` subset or a
  transient total-fetch failure can't skip postings next run. `--dry-run` never
  advances. A missing/unparseable per-job timestamp is **kept** (never silently
  dropped ‚Äî over-collect is recovered by dedupe; under-collect is data loss).

**Incremental scoring:** New records scored automatically after
collection. `ApplicationRecord` created with `application_status:
"new"` and `application_decision: "pending"`.

**Daily digest:** `digest.py` generates a summary of new roles
above a fit threshold.

```bash
python digest.py --min-fit 6 --since yesterday
```

Output: terminal summary + optional export to `corpus/digest_{date}.md`

**New source candidates** (evaluate before adding):
- LinkedIn job alerts via email ‚Üí parse digest emails
- RSS feeds from company career pages
- Google Alerts for target companies + "hiring"

Done:
- Incremental collection in `collect.py` (cursor + `--full`); client-side filter
  in `collectors/base.passes_cursor`; per-collector `SUPPORTS_INCREMENTAL`.
- `cli/digest.py` ‚Äî daily digest CLI (run as `python -m cli.digest`). Joins the
  latest scored records (`ApplicationRecord`) with JD + sidecar + the activity-log

### 9.4 ‚Äî Architecture

**Static index approach:**

`python -m cli.stats --input "corpus/validated/validated_*.jsonl" --export-index`
generates `corpus/index.json`. The UI reads this one file. No API, no
database, no additional backend service.

`index.json` is **not** a bare JDRecord array (as an earlier draft of this
section sketched). The UI needs scoring (`fit_label` / `fit_score` /
`priority_score` / `fit_label_reason` / `requirement_gaps`) and **live workflow
status**, none of which a JDRecord carries. So the index is the **same join the
tracker performs** (CLAUDE.md deviation 23): latest `ApplicationRecord` per
`job_id` ‚®ù `JDRecord` extraction ‚®ù metadata sidecar ‚®ù activity-log projection,
denormalised one row per **scored** job. The file is an object:

```json
{
  "schema_version": "1.3",
  "jdrecord_schema_version": "1.2",
  "generated_at": "‚Ä¶Z",
  "stats": { "total", "by_fit_label", "by_application_status",
             "fit_score_distribution", "cost_to_date_usd" },
  "records": [ { ‚Ä¶joined row‚Ä¶ } ]
}
```

`stats` is embedded (not a second file) so the single mounted `index.json` is
self-contained ‚Äî the UI container mounts only that file, never `stats.json` or
the corpus. `cost_to_date_usd` is summed from `corpus/stats.json` at export time.
See deviation 27.

The UI is a static HTML page with vanilla JavaScript (`ui/index.html` +
`ui/app.js` + `ui/style.css`). Deployable anywhere. No build step, no framework,
no external CDN. Strictly read-only: it never POSTs, writes, or invokes a CLI.

**Filters available:**

- priority_score (min/max)
- application_status (multi-select)

- domain (multi-select)

- seniority (multi-select)
- company_stage (multi-select)
- date_first_seen (range)

**Docker service:**

### 10.4 ‚Äî Architecture

**Stack:**
```
React (Vite)                  frontend ‚Äî replaces ui/ static files
FastAPI                       thin backend ‚Äî mediates all writes, serves index.json
corpus/activity_log.jsonl     workflow state ‚Äî unchanged, appended via API
corpus/annotations.jsonl      scoring flags ‚Äî new, appended via API
corpus/index.json             read cache ‚Äî regenerated by stats --export-index
JSONL pipeline                unchanged throughout
```

**No database.** SQLite deferred ‚Äî add when complex queries across 1000+
records, multiple concurrent users, or transactions are needed. None apply
now.

**API surface ‚Äî task-oriented, not event-oriented:**

Status updates are 90% of UI writes. The API speaks the domain language
the UI uses. The event model (`activity_log.jsonl`) stays underneath ‚Äî
each task endpoint validates and appends to the appropriate JSONL file.

```
# Public ‚Äî no auth required
GET  /api/index              serves corpus/index.json (the joined read model)
GET  /api/capabilities       {write_configured, write_unlocked}
GET  /api/health             {status, records, last_indexed}

# Auth
POST /api/unlock             validates key, sets HttpOnly capability cookie
POST /api/lock               clears capability cookie

# Workflow writes ‚Äî require valid capability cookie
POST /api/status             {job_id, status, notes?}
                             appends status event to activity_log.jsonl
                             validated against APPLICATION_STATUS vocab

POST /api/note               {job_id, text}
                             appends note event to activity_log.jsonl
                             pure note ‚Äî no status change

POST /api/title              {job_id, title}
                             appends title event to activity_log.jsonl
                             display override only, never scored

POST /api/outcome            {job_id, outcome, notes?}
                             appends outcome event to activity_log.jsonl
                             validated against OUTCOME vocab. The UI also POSTs
                             /api/status to move the lane (rejected_* ‚Üí "rejected",
                             withdrew/offer_declined ‚Üí "archived"); the rejection
                             stage is auto-derived from current status (applied‚Üí
                             post_screen, interviewing‚Üíinterview, offer‚Üífinal) but
                             editable. Added post-M2 from use ‚Äî the OUTCOME model +
                             CLI `--outcome` existed; only the HTTP/UI surface was
                             missing (deviation 32). application_date is derived
                             from the earliest applied event; the UI shows
                             age-since-applied + a "stale" flag past 21 days.

# Annotation writes ‚Äî require valid capability cookie
POST /api/annotations        {job_id, annotation_type, field,
                              observed, expected, reason}
                             appends to corpus/annotations.jsonl
                             validated against ANNOTATION_TYPE frozenset
```

**Why task-oriented over generic `POST /api/events`:**
- Each endpoint validates only what it needs ‚Äî `/api/status` validates
  against `APPLICATION_STATUS`; `/api/note` validates non-empty text.
  No shared payload shape all callers must get right.
- FastAPI auto-generates self-documenting OpenAPI ‚Äî four clear operations,
  not one generic event emitter with a payload schema to decode.
- Frontend code is clean ‚Äî `api.shortlist(jobId)` not
  `api.emitEvent({job_id, event: 'status', value: 'shortlisted'})`.
- Error messages are specific and actionable per endpoint.
- `POST /api/note` is separate from `POST /api/status` with optional notes
  because a pure note (no state change) is a different intent ‚Äî the frontend
  never has to reason about which combination to send.

**FastAPI is thin.** Imports `cli.track` for event validation and appending.
Imports `models.record` for vocab constants. Never calls the scorer,
labeller, or any pipeline stage. A write request is: validate ‚Üí append
‚Üí return 200.

### 10.5 ‚Äî Security model detail

Following cv-tailor D-38/D-39 exactly. Copy and adapt `api/security.py`
from cv-tailor ‚Äî same HMAC + HttpOnly cookie pattern, proven implementation.

**Unlock flow:**
1. Owner clicks any write action (status, note, title, annotation)
2. If not unlocked: UI shows unlock dialog (password field ‚Äî never
   persisted in React state, localStorage, or a readable cookie)
3. UI POSTs key to `POST /api/unlock`
4. Backend validates against `FULL_MODE_KEY`. Success ‚Üí signed HttpOnly
   cookie. Failure ‚Üí 401, user stays read-only.
5. Subsequent writes rely on the cookie; key never re-sent per action.
6. Unlocked session shows a small "owner" indicator with a lock affordance.

**`GET /api/capabilities` drives UI rendering:**

| `write_configured` | `write_unlocked` | UI behaviour |
|---|---|---|
| false | false | Write controls hidden entirely ‚Äî no dead buttons |
| true | false | Write controls visible, click ‚Üí unlock dialog |
| true | true | Write controls enabled, owner indicator shown |

**Fail-closed:** no `FULL_MODE_KEY` ‚Üí all write endpoints 403, public
deployment is clean read-only (matching Phase 5 experience for visitors).

## 11. Phase 7 ‚Äî Fine-Tuned Analyser (Future Enhancement)

This phase is **explicitly deferred**. It becomes Project 5 when the
corpus is large enough to justify fine-tuning (target: 500+ labelled
records with human-validated eval set).

**What it does:** Replaces the rule-based scoring engine (Phase 2)
with a fine-tuned model that predicts fit_score and blocking_constraints
from raw JD text. The corpus built in Phase 1 is the training data.
The eval set from Phase 1 is the evaluation set.

**Why deferred:** Fine-tuning a domain-specific model on <200 records
produces marginal gains over a well-designed rule-based scorer. The
rule-based scorer in Phase 2 is good enough to be useful immediately.
Fine-tuning becomes valuable when the corpus is large and the
rule-based scorer's failure modes are well-understood.

**Handoff to Project 5:**
- `corpus/finetune_export/export_train_*.jsonl`
- `corpus/finetune_export/export_eval_*.jsonl`
- `CORPUS_FINDINGS.md` ‚Äî schema, labelling rules, failure modes
- `corpus/stats.json` ‚Äî cost baseline
- `corpus/annotations.jsonl` ‚Äî field-level scoring disagreements from
  Phase 6 (ground-truth correction signals for fine-tuning)

## 13. Architecture Decisions

### 13.1 ‚Äî Job Radar as one product, not separate projects

**Decision:** Phases 1‚Äì6 are one product. Phase 7 is Project 5.

**Rationale:** The corpus builder alone is not useful to Michel the
job seeker. The job search tool needs the corpus. Building them
separately creates a handoff problem and two incomplete products.
One product with phases produces value incrementally.

### 13.2 ‚Äî Rule-based scoring before fine-tuning

**Decision:** Phase 2 is rule-based. Fine-tuning is Phase 7 / Project 5.

**Rationale:** 200 labelled records is insufficient for meaningful
fine-tuning of a domain-specific scorer. A well-designed rule-based
scorer on a structured schema produces comparable results immediately,
without GPU, without training cost, and without the data requirements
of fine-tuning. Fine-tuning adds value after the corpus exceeds 500
records and the rule-based scorer's failure modes are documented.

### 13.3 ‚Äî Three-layer schema separation

**Decision:** JDRecord (extraction) / JobPosting (product) /
ApplicationRecord (annotation) are separate dataclasses.

**Rationale:** Mixing extraction labels with personal judgement
contaminates the training corpus. Mixing product state with extraction
creates migration problems when the product evolves. Three layers
with clear ownership (Claude / system / Michel) keep each concern
clean.

### 13.4 ‚Äî Static index for UI

**Decision:** Phase 5 static UI reads `corpus/index.json`, not a live
API. Phase 6 interactive UI keeps `index.json` as the read model but
adds a FastAPI backend for writes.

**Rationale:** The read model (a pre-built joined index) is simpler,
faster to render, and consistent with "files all the way down." A
live API adds operational complexity only worth taking on when write
capability is needed ‚Äî which Phase 5 didn't need and Phase 6 does.

### 13.5 ‚Äî CLI writes, UI reads (Phase 5)

**Decision:** Phase 5 ‚Äî all writes go through CLI. UI never modifies
state.

**Rationale:** Preserves the CLI as the single source of truth.
Prevents accidental writes from the UI corrupting JSONL files.
Keeps the UI stateless and deployable anywhere.

### 13.6 ‚Äî Task-oriented API, not generic event emitter (Phase 6)

**Decision:** Phase 6 FastAPI exposes four task-oriented write
endpoints (`/api/status`, `/api/note`, `/api/title`,
`/api/annotations`) rather than a single generic `/api/events`.
The activity log event model remains underneath.

**Rationale:** Status updates are 90% of UI writes. The API should
speak the domain language the UI uses, not the infrastructure language
of the event log. Four specific endpoints validate only what they need,
generate self-documenting OpenAPI output, and produce specific error
messages. Frontend code reads as `api.shortlist(jobId)` not
`api.emitEvent({job_id, event: "status", value: "shortlisted"})`.
`/api/note` is separate from `/api/status` with optional notes because
a pure note (no state change) is a different intent ‚Äî the frontend
never has to reason about which payload combination to send.

### 13.7 ‚Äî "CLI writes, UI reads" refined for Phase 6

**Decision:** Phase 6 supersedes ¬ß13.5 for workflow state and
annotations only. Writes that produce pipeline artifacts (collection,
labelling, scoring, validation) remain CLI-only. Writes that record
human judgement (workflow state, scoring annotations) are available
via both CLI and FastAPI, validated identically by the same
`models.record` constants.

**Rationale:** The read-only constraint was correct during construction.
Phase 5 delivered it correctly. But it creates daily friction: every
status update and annotation requires a terminal. The FastAPI layer
appends to the same JSONL files the CLI appends to ‚Äî the data model
is unchanged, the source of truth is unchanged, the CLI remains a
valid write path. No database is introduced. "Files all the way down"
is preserved.

## 15. Interview Narrative

**One product, multiple learning layers.** Job Radar is not a demo
project. It is a system Michel uses daily for his job search. This
means it has real requirements, real failure modes, and real
constraints ‚Äî not invented ones. That authenticity shows in the
architecture decisions.

**Schema as the hardest problem.** The extraction schema was validated
on 10 real JDs before any code was written. It changed 10 times.
The most significant changes ‚Äî splitting required_skills into four
fields, separating extraction from annotation, adding delivery_motion
as a first-class dimension ‚Äî were not foreseeable from first principles.
They emerged from practice. That is how real data engineering works.

**Three-layer schema separation.** The JDRecord / JobPosting /
ApplicationRecord split demonstrates understanding of the difference
between what data says, what a product does with it, and what a user
thinks about it. Most candidates conflate these. The ability to name
the boundary and enforce it structurally is a differentiator.

**Incremental value delivery.** Phase 1 produces a working corpus.
Phase 2 produces actionable fit scores. Phase 3 replaces a
spreadsheet. Each phase is independently valuable. This is how
production systems get built ‚Äî not in one big release.

**Cost awareness from the start.** Batch API chosen in Phase 1 before
a single labelling run. Cost tracked per run. Fine-tuning deferred
until the corpus justifies it. These are the decisions of someone
who has thought about production constraints, not someone building
a demo.
